1.3 Security Considerations for Real-Money Transactions
Non-negotiable security baseline:

Credential encryption at rest — Broker API keys/secrets encrypted with Fernet; master key from MASTER_ENCRYPTION_KEY env var (minimum 32 bytes, generated via openssl rand -base64 32).
Argon2id password hashing — t=3, m=64MB, p=4; no MD5/SHA1/bcrypt fallbacks.
TOTP mandatory for live mode — cannot enable live trading without 2FA verified.
HTTPS mandatory — Caddy auto-provisions Let's Encrypt certs; HTTP refuses to serve sensitive endpoints.
Session cookies — HttpOnly, Secure, SameSite=Strict, 24h idle / 7d absolute expiration.
CSRF protection — Double-submit cookie pattern on all mutating endpoints.
Rate limiting — slowapi enforces 5 login attempts per 15 min, 100 req/s general API.
Idempotency keys — UUID v7 on every order; duplicate submissions rejected at DB level (unique constraint).
Append-only audit log — Enforced by SQL trigger blocking DELETE on audit_log table.
Order reconciliation — Every 30 seconds the worker compares local order/position state with broker truth; discrepancies alert the user.
Kill switch — Multiple trigger paths (UI button, API endpoint, Unix signal SIGUSR1, daily loss breach); flattens positions and halts engine.
Pre-trade risk checks — Every order passes through max value, max positions, price sanity, rate limit, kill-switch-active checks before hitting broker.
Secrets never in logs — Log formatter redacts fields matching api_key|secret|token|password|totp.
No telemetry — Zero outbound calls except to configured brokers and optional notification endpoints.
Dependency scanning — pip-audit and npm audit in CI; CRITICAL CVEs block builds.

3. User Workflow Documentation
3.1 Authentication and Authorization Flow
Single-user system — only one admin role exists.

User navigates to domain → Caddy serves HTTPS → Next.js serves /login
User enters username + password → POST /api/auth/login
API validates rate limit (≤5 attempts / 15 min / IP) → Argon2id verify → issues 2FA-pending token (5-min TTL)
Browser redirects to /login/2fa
User reads TOTP from authenticator → enters 6 digits → POST /api/auth/2fa
API validates TOTP via pyotp (30s window, ±1 skew) → issues full session cookie (HttpOnly, Secure, SameSite=Strict, 24h idle / 7d absolute)
Redirect to /dashboard
Every subsequent API request includes session cookie; API middleware validates signature, expiration, user existence; refreshes idle timer
On logout or expiration → session invalidated server-side → redirect to /login
3.2 Zero-to-First-Real-Trade Workflow (17 Steps)
This is the complete path from clean install to first real-money order.

Phase A: Prerequisites (external, before install)
User opens a broker account with one of: Zerodha, Dhan, Upstox, Fyers, Angel One
User completes broker KYC and funds the account via broker's own channels
User subscribes to the broker's API (e.g., Zerodha Kite Connect ₹2000/month; most others free)
User creates an API app in the broker's developer console, noting API Key, API Secret, and setting Redirect URL to https:///api/brokers//callback
Phase B: Installation (local, ~10 minutes)
User provisions a Linux host (Ubuntu 22.04+, 2 vCPU / 4 GB RAM / 20 GB disk) with Docker
User runs:
git clone https://github.com/lohi-trade/lohi-trade-oss
cd lohi-trade-oss
cp .env.example .env
# Edit .env: DOMAIN, MASTER_ENCRYPTION_KEY (openssl rand -base64 32), DB_MODE=sqlite
docker compose up -d
User opens https:// → lands on /setup
Phase C: First-Run Setup (~5 minutes)
User completes 5-step setup wizard (license → admin account → 2FA → defaults → completion)
Phase D: Broker Connection (~5 minutes)
User logs in → navigates to /settings/brokers → clicks [+ Add Broker]
User picks broker, confirms prerequisites, enters credentials → [Test Connection] succeeds → [Save & Connect]
Engine establishes WebSocket to broker's market-data endpoint; scheduled job registers for daily token refresh (Zerodha/Angel One)
Phase E: Paper Trading Validation (≥1 trading session, strongly enforced)
User navigates to /strategies → picks bundled educational strategy (e.g., SMA crossover) → [Configure] → selects a symbol → [Save & Start]
Strategy runs in PAPER mode throughout at least one trading session; user verifies expected behavior
User reviews /positions, /orders, /audit to confirm everything behaves as expected
Phase F: Risk Configuration (~3 minutes)
User navigates to /settings/risk → reviews defaults → adjusts if needed (e.g., tightens max order value to comfort level)
Phase G: Live Mode Activation (~2 minutes, deliberate friction)
User scrolls to "Enable Live Trading" section → clicks toggle
Modal Step 1: User checks all 6 confirmations (system blocks if broker not configured or no paper sessions logged)
Modal Step 2: User re-enters password + current TOTP
Modal Step 3: User types ENABLE LIVE TRADING exactly → [Enable Live Trading]
Dashboard banner turns red; notification fires; audit log entry created
Phase H: First Real Order
Two paths:
Manual: User navigates to /trade (or ⌘B) → fills ticket → [Review Order] → red warning banner → [Confirm Order] → idempotency key assigned → pre-trade risk checks pass → broker adapter calls broker REST API → order ID returned → toast notification → order visible in /orders with SUBMITTED status → fill event received via WebSocket → status becomes FILLED → position appears in /positions
Strategy: User starts a previously paper-validated strategy in LIVE mode → strategy generates signal → engine submits order identically to manual flow
3.3 Fund Deposit and Withdrawal Processes
Critical: LOHI-TRADE never touches funds.

Deposits: User logs into broker's own app/website (Zerodha Kite, Dhan app, Upstox, etc.) → deposits funds via broker's UPI/netbanking/NEFT channels → funds reflect in broker account.
Withdrawals: User logs into broker's app → initiates withdrawal to linked bank account per broker's T+1/T+2 settlement rules.
LOHI-TRADE's role: Display available margin/funds queried live from broker's API (read-only) on order ticket and dashboard. No payment gateway. No bank account integration. No fund movement logic. This is a deliberate architectural choice to avoid SEBI intermediary obligations on the self-hosting user.
3.4 Trade Execution Workflow (Internal)
For every order, automated or manual:

Order request received (from UI or strategy)
Idempotency key generated (UUID v7)
Pre-trade risk checks executed in sequence:
kill_switch_active → reject if true
trading_mode == LIVE check (or reject if attempting live order in paper mode)
max_order_value check against order notional
max_open_positions and max_open_positions_per_strategy checks
price_sanity check against last tick (±5% default)
max_orders_per_minute rate-limit check
Product-specific checks (short selling allowed? F&O allowed? options writing allowed?)
Session-window check (within allowed hours?)
If any check fails → order rejected → reason logged → user notified → audit log entry
If all pass → order persisted to orders table with status PENDING → broker adapter invoked
Broker adapter handles protocol specifics (REST POST to /orders/regular for Zerodha, /orders for Dhan, etc.)
Broker response processed:
Success → status SUBMITTED → broker order ID stored → WebSocket event to UI
Failure → status REJECTED → error reason stored → WebSocket event → user notified
Broker sends order update events via WebSocket (order acknowledged, partially filled, filled, cancelled, rejected)
Engine receives event → updates orders and fills tables → recomputes positions → broadcasts to UI
Worker reconciliation job every 30s calls broker's GET /orders and GET /positions → compares with local state → flags discrepancies
3.5 Account Management Procedures
Password change: /settings/profile → requires current password + new password + 2FA re-entry
TOTP reset: requires password + 12-word recovery phrase from initial setup
Recovery phrase lost + password forgotten: CLI-only recovery: docker compose exec api lohi reset-password  (requires shell access to host → assumed trusted)

7.3 Error Handling and Rollback Procedures
Order-level error handling:

Error class	Handling
Broker API 5xx	Retry with exponential backoff (3 attempts, 1s/2s/4s); if still failing, mark order FAILED, notify user, do not retry automatically
Broker API 4xx (client error)	No retry; mark REJECTED with reason; notify user
Network timeout	Retry once; if still failing, query broker's order book to check if order actually placed (by idempotency key); reconcile state
Rate limit breach	Queue the order; retry after broker's reset window
Duplicate idempotency key detected	Reject the new submission; return existing order ID
System-level error handling:

Error class	Handling
Broker WebSocket disconnect	Exponential backoff reconnect (1/2/4/8/60s cap); subscription list persisted and re-registered
DB connection failure	Health check fails; Docker Compose restart policy kicks in (restart: unless-stopped); engine pauses order submission until DB healthy
Engine crash	Supervisor restarts engine; on startup, engine reads active strategies from DB and resumes; open orders reconciled with broker
Kill switch triggered	All strategies stopped immediately; open orders cancelled via broker API; all positions flattened (configurable); notification fired; manual reset required
Corrupt DB	Alembic migration detects incompatible schema; lohi doctor provides diagnostic; restore from backup using lohi restore 
Rollback procedures:

Bad strategy deployment: Stop strategy via UI → orders placed by strategy remain but no new ones → manually close positions via Positions page or bulk [Close All].
Bad configuration change: Audit log shows what changed by whom/when; lohi config rollback --steps=1 CLI reverts last N changes (config only, not orders).
Accidental live-mode enable: Toggle off in /settings/risk → confirmation modal → mode returns to paper → running strategies continue but subsequent orders routed to paper simulator.
Complete disaster: Restore latest backup via lohi restore; reconcile with broker's authoritative state for positions/orders; manually resolve any discrepancies.
Post-incident review (recommended):

lohi doctor --since= generates a diagnostic bundle (redacted) with logs, recent audit entries, and system state
Bundle saved locally only; user decides whether to share for support
Documentation includes a "post-incident checklist" in the Going Live guide
Critical operational principle: The system is designed so that every automated action that risks capital can be interrupted by a human within 2 clicks, and every state change is reversible except the trades themselves (which require market action to unwind).
